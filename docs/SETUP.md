# Setup Guide

This guide walks through everything you need before running `conf.py build`
for the first time.

## 1. Google Cloud project and OAuth credentials

The build script reads Google Sheets and Google Drive using OAuth 2.0
(a Desktop app client, not a service account).

### Create a project and enable APIs

1. Go to [console.cloud.google.com](https://console.cloud.google.com)
2. Create a new project (e.g. `my-conf-schedule`)
3. Enable these two APIs:
   - **Google Sheets API**
   - **Google Drive API**

### Create OAuth credentials

1. In the project, go to **APIs & Services → Credentials**
2. Click **Create credentials → OAuth client ID**
3. Application type: **Desktop app**
4. Name it anything (e.g. `schedule-builder`)
5. Download the JSON file and save it as `credentials.json` in the repo root

`credentials.json` is gitignored — never commit it.

### First-run browser authentication

On the first `conf.py build` (or any script that calls `auth.sheets_service()`
or `auth.drive_service()`), a browser window opens for you to authorize the
app. After you click "Allow", a `token.json` file is cached in the repo root
and reused on subsequent runs.

`token.json` is gitignored — never commit it.

### Add yourself as a test user (if needed)

If your Cloud project is in "Testing" mode (OAuth consent screen not published),
you must add your Google account as a test user:
**APIs & Services → OAuth consent screen → Test users → Add users**

## 2. Google Sheet structure

The build script reads three tabs from a single Google Sheet. The tab names
must match `ACAL_TALKS_TAB`, `BANTO3D_TALKS_TAB`, and `POSTER_TAB` in
`config.py`.

### Talks tabs (ACAL and co-conference)

Each tab is a grid where:
- **Column A** is the time/session label
- **Columns B onward** are parallel track columns
- Cells contain one talk per cell; the first line is the title, subsequent
  lines are presenters (one per line), optionally followed by metadata lines
  starting with `Location:`, `Zoom:`, `Chair:`, `Tech:`, etc.

Session header rows use a specific format that the parser recognizes (see
`schedule/parse.py` → `parse_acal()` for the exact logic).

### Poster tab

Each row is one poster. Expected columns (by header name):
- A column for the poster number/label
- Columns for presenter name(s), title, modality (In-Person / Remote), and
  any notes

### Sheet ID

Copy the ID from the URL:
```
https://docs.google.com/spreadsheets/d/THIS_PART/edit
```
Paste it into `SHEET_ID` in `config.py`.

## 3. Google Drive folder layout

```
Presentations folder (PRESENTATIONS_FOLDER_ID)
├── Abstracts/              ← all abstract PDFs (DRIVE_FOLDER_ID)
│   ├── CONF001.pdf
│   ├── CONF002.pdf
│   └── ...
├── Plenaries/              ← plenary speaker PDFs (PLENARY_FOLDER_ID)
│   ├── SpeakerA.pdf
│   └── ...
├── Session 1A/             ← one subfolder per session (session_folders.csv)
├── Session 1B/
│   └── ...
└── CONF-schedule.pdf       ← stable PDF file (PDF_DRIVE_FILE_ID)
```

### Creating session folders

For each session in the schedule, create a Drive subfolder inside the
Presentations folder. Record the folder ID and URL in `session_folders.csv`:

```
session,folder_id,url,tab,row,col
Session 1A,FOLDER_ID,https://drive.google.com/drive/folders/FOLDER_ID,CONF Talks,,
```

The `tab`, `row`, and `col` columns are auto-synced on each build run —
you only need to fill in `session`, `folder_id`, and `url`.

### Abstract PDF naming convention

Name each PDF consistently so the fuzzy matcher can link it to schedule
entries. The ACAL 57 convention was `CONF001.pdf`, `CONF002.pdf`, etc.,
matching submission IDs in the abstract tracking xlsx.

## 4. GitHub Pages repository

The schedule publishes to a dedicated GitHub repository with Pages enabled.

### Create the repo

1. Create a new public GitHub repo (e.g. `username/my-conf-schedule`)
2. Go to **Settings → Pages**
3. Set Source to **Deploy from a branch**, branch `main`, folder `/root`
4. The schedule will be live at `https://username.github.io/my-conf-schedule/`

### Update config.py

In `config.py`, set:
```python
GITHUB_REPO = 'username/my-conf-schedule'
PAGES_URL   = 'https://username.github.io/my-conf-schedule/'
```

## 5. gh CLI authentication

The build script uses `gh` (GitHub CLI) to get an authentication token for
the GitHub Contents API.

Install `gh`: https://cli.github.com

Then authenticate:
```bash
gh auth login
```

Choose **GitHub.com**, **HTTPS**, and follow the prompts.

Verify it works:
```bash
gh auth status
```

---

## 6. wkhtmltopdf (for PDF generation)

The PDF schedule is generated by `wkhtmltopdf` rendering the print HTML.

Install on macOS:
```bash
brew install wkhtmltopdf
```

Install on Ubuntu/Debian:
```bash
sudo apt-get install wkhtmltopdf
```

Verify:
```bash
wkhtmltopdf --version
```

```bash
python3 -m venv .venv

```

Key packages: `google-api-python-client`, `google-auth-oauthlib`,
`openpyxl`, `rapidfuzz`.

## 8. Validation checklist

Run `conf.py doctor` to verify everything before your first build:

```bash
.venv/bin/python3 conf.py doctor
```

It checks:
- All `config.py` values filled in (no `'FILL_IN'` remaining)
- `credentials.json` exists
- Google Sheets API accessible (can reach the sheet)
- Google Drive API accessible (can list the abstract folder)
- `gh` CLI authenticated
- `wkhtmltopdf` in PATH
